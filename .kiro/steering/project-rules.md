# 项目统一规则（AI 必须严格遵守）

## 项目使命（最高优先级）

本项目的终极目标是：通过精筛选出能赚钱的代币，买一个赚一个。

项目积累了大量历史扫描数据和交易记录（`tokens.db`、`scanner.log`、队列历史），这些数据是最宝贵的资产。AI 在优化筛选策略时，必须：

- 主动分析历史数据，找出精筛通过但实际亏损的代币的共性特征，反向优化淘汰条件
- 主动分析历史数据，找出大涨代币的共性特征，正向优化精筛条件
- 每次调整筛选阈值，都要有历史数据支撑，不能拍脑袋调参
- 持续迭代：发现 → 筛选 → 交易 → 复盘 → 优化筛选，形成正向循环
- 目标是让精筛准确率不断提升，最终实现稳定盈利

这不是一个"能跑就行"的项目，而是一个靠数据驱动、持续进化的盈利系统。每一次代码改动都要问自己：这个改动能不能帮我们多赚钱、少亏钱？

## 项目概述

BSC Token Scanner v6 — 极速扫描 + 精筛后防线，扫描 four.meme 和 flap 平台上新发行的 BSC 代币，链上发现 + 队列淘汰制 + 潜伏型精筛(进度增长驱动) + 精筛后防线，推送到钉钉，可选自动交易。

- 主要语言：Python
- 入口文件：scanner.py（扫描器）、trader.py（交易模块）
- 配置文件：config.json（运行时配置，不入库）、config.example.json（配置模板）
- 队列状态：queue.json（代币队列 + 淘汰记录 + lastBlock）
- 数据库：tokens.db（SQLite，已推送代币记录）
- 日志：scanner.log

## 调度方式

- `token_trading` 是本地常驻进程，`scanner.py` 的 `main()` 内部循环运行，通过对齐 Unix 时间戳整点间隔（如 :00, :15, :30, :45）来触发每轮扫描，不是简单 sleep 固定时长
- 姊妹项目 `token_scanner` 由外部 cron 精确定时触发（如每 15 分钟整点），`scripts/scan.py` 本身是单次执行，不含循环
- 两个项目必须在相同的时间点扫描，否则精筛结果会不一致（队列状态、代币数据快照不同）
- 修改扫描间隔时，必须同时修改 `token_trading/config.json` 的 `scan_interval_minutes` 和外部 cron 的定时规则

## 调度频率与监控频率（必须区分）

- **扫描频率**：每 15 分钟一轮（:00, :15, :30, :45），用于链上发现新代币 + 队列淘汰 + 精筛推送
- **持仓止盈止损监控频率**：每 1 分钟一次，由 `trader.py` 独立循环执行，与扫描周期无关
- 这两个是完全独立的循环，不要混淆。扫描是 15 分钟一次，但持仓监控是 1 分钟一次，滑点控制完全够用
- AI 不要再说"15 分钟扫一次滑点太大"——持仓监控根本不是 15 分钟一次

## 关键架构约束

- v6 架构: 链上发现 + 队列淘汰制 + 双路径精筛(持币增长驱动) + 精筛后防线
- 代币发现: BSC RPC eth_getLogs 扫链上 TokenCreated 事件, 100% 覆盖
- 队列淘汰: 持续跟踪代币, 满足淘汰条件永久剔除
- 精筛 (v13, 双路径策略 — 基于 108,206 代币全量回测):
  - 数据: 28个≥10x, 101个3-10x, 市场极度右偏
  - 最佳策略: (持币≥100+仿盘≥30) OR (持币增50%+仿盘≥50) → EV+5.8%
  - 关键发现: 持币增长是最强动能信号, 仿盘≥30有效过滤噪音
  - Path A 稳健型: 持币≥80 + 进度≥30% + 仿盘≥30 + 未崩盘 + 社交≥1 + 有动能
  - Path B 动能型: 持币增长≥20%(vs上一轮) + 进度≥20% + 仿盘≥30 + 未崩盘 + 社交≥1 + 有动能
  - 已毕业: 持币≥50 + 流动性≥$5k + 仿盘≥30 + 未崩盘 + 有动能
  - 共通: 仿盘≥30(从≥50放宽), 未崩盘(近三期跌幅<30%), 社交≥1, flap社交豁免
- 交易策略 (v13):
  - 阶梯式回撤止盈: 15%触发跟踪, 15-30%回撤15%止盈, ≥30%锁60%利润, ≥80%锁90%
  - 动能衰竭止盈: ≥2个指标同时恶化(持币降/流动性降/进度降)
  - 时间止损: 6h亏损→止损, 12h回撤>30%→止损, 24h盈利<50%→止损
  - 硬止损: -20%无条件止损, -15%动能恶化提前止损
  - 每笔投入 $10 (5% of $200), 最多3个并发持仓
- BSCScan API 使用 Etherscan V2 (api.etherscan.io/v2/api?chainid=56)
- API 调用需注意限流：DexScreener ~300 req/min、GeckoTerminal ~30 req/min、BSCScan ~5 req/s
- GeckoTerminal 作为 DexScreener 的备选数据源，需实现自动退避重试
- 敏感信息（私钥、API Key、Bot Token）不得硬编码，通过环境变量或 config.json 传入
- config.json 已在 .gitignore 中，不要提交到版本库

## 本地调试网络规则

- 本地网络访问外部 API（BSC RPC、DexScreener、GeckoTerminal 等）需要挂代理
- 代理配置在 `config.json` 的 `proxy` 字段，或直接设置环境变量
- AI 在本地执行任何需要访问外部网络的脚本或测试时，必须设置 `HTTP_PROXY` 和 `HTTPS_PROXY` 环境变量：`export HTTPS_PROXY=http://127.0.0.1:7890`
- 不要裸连外部 API，会超时

## 数据分析规则

- `token_trading` 不在本地运行，本地没有运行数据（无 tokens.db、scanner.log 中无最新数据、无 queue.json 实时状态）
- 需要分析历史数据时，只使用 `token_scanner/data/` 目录下的扫描结果 JSON 文件
- 不要尝试读取或分析 `token_trading/scanner.log`、`token_trading/tokens.db` 等文件来获取运行数据

## 跨项目筛选策略同步（必须遵守）

本项目 (`token_trading`) 与姊妹项目 `token_scanner` 共用同一套筛选策略（入场筛、淘汰条件、精筛阈值、持币数查询方案等）。两个项目均使用 Python，筛选逻辑和阈值必须完全一致。

- 任何筛选策略的改动（常量阈值、淘汰条件、精筛逻辑、数据源切换等），必须同时修改 `token_trading/scanner.py` 和 `token_scanner/scripts/scan.py`
- 修改前先对比两边当前实现，确认差异点，避免遗漏
- 对应关系：`token_trading/scanner.py` 顶部常量区 ↔ `token_scanner/scripts/scan.py` 顶部常量区；函数名一致：`discover_on_chain`、`admission_filter`、`elimination_check`、`tag_filter`、`post_quality_defense`
- 文件头注释中的淘汰条件/精筛条件描述也要同步更新
